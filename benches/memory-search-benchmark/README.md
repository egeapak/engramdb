# Memory search & lookup benchmark

An end-to-end benchmark that drives **real `engramdb serve` MCP sessions** (stdio
JSON-RPC) through the `query` (search) and `get` (lookup) tools, and measures how
the **shared embedding daemon** compares to **in-process** model loading across a
latency × memory × concurrency matrix.

## What it measures

Matrix dimensions:

| dimension | values | meaning |
|---|---|---|
| `execution` | `in_process`, `daemon` | each session loads its own ONNX embedding model, vs. routing inference to the shared daemon (model resident once machine-wide) |
| `target` | `local`, `global` | a single project store vs. the cross-project global store |
| `ops/iter` | 1, 2, 4 | mixed search + lookup operations per iteration (search, lookup, search, lookup) |
| `sessions` | 1, 2, 4 | parallel MCP sessions running the same workload |

Metrics per cell:

- **session startup (ms)** — spawn → MCP `initialize` complete.
- **time-to-first-result (ms)** — spawn → first search result. In-process this
  includes the one-time embedding-model load (the cost the daemon amortizes);
  the model loads in a background warmup task, so the first embedding query
  blocks on it. Cleanest at `sessions=1` (no overlap between sessions'
  concurrent background loads).
- **warm op p50 / p95 (ms)** — steady-state per-op latency after warmup.
- **iter wall p50 (ms)** — wall time for all parallel sessions to finish one
  iteration.
- **throughput (ops/s)**.
- **RSS** — resident memory per session process, plus the shared daemon, plus
  the total. This is where the daemon's model-sharing shows up: in-process RSS
  scales ~linearly with session count (one model each); the daemon keeps a
  single resident model.

## How execution mode is controlled

The MCP server ignores `ENGRAMDB_IN_PROCESS`; the real lever is
`daemon.enabled` in the store's `.engramdb/config.toml` (the master switch in
`resolve_providers`). `bench.py` writes `enabled = false` for the in-process
phase (and stops any daemon) and `enabled = true` for the daemon phase (and
starts/pre-warms a daemon), since config is reloaded per request.

## Files

- `gen_data.py` — generates the dataset: 10 themed projects × 10 memories each
  (spanning all 8 memory types, with realistic scopes/tags so semantic search
  has signal) + 10 org-wide global memories.
- `bench.py` — the matrix harness (minimal MCP stdio client, /proc RSS sampling,
  daemon lifecycle + config toggling).
- `report.py` — renders `results.json` into `RESULTS.md`.
- `run_all.sh` — orchestrates everything; isolates all EngramDB state under a
  scratch `ENGRAMDB_DATA_DIR`/`ENGRAMDB_CONFIG_DIR` so it never touches real data.

## Running

```bash
# from the repo root, with a release binary built
benches/memory-search-benchmark/run_all.sh
```

Environment knobs:

- `BENCH_SCRATCH` (default `/tmp/engram-bench`) — scratch dir for all state.
- `BENCH_ITERS` (default `8`) — timed iterations per cell.
- `BENCH_REGEN=1` — force dataset regeneration.
- `ENGRAM_BIN` — path to the `engramdb` binary (default `target/release/engramdb`).

On the Claude Code web sandbox the binary must be built with the ONNX runtime
workaround (`ORT_STRATEGY=system ORT_LIB_LOCATION=/tmp/ort-lib`) and the
embedding model staged under `~/.cache/engramdb/models/` — see the repo
`CLAUDE.md`. At run time only the staged model cache is needed.
