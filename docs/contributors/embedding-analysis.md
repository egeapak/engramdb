# Embedding strategy analysis: chunking, models, and field composition

*2026-07-20 · benchmark assets: `examples/embed_matrix.rs`, `examples/data/embed_eval.json`, results snapshot `examples/data/embed_matrix_results.json`*

This document reports a controlled offline study of EngramDB's embedding
pipeline: how much chunking strategy, embedding-model choice, and the
*composition of the embedded text* each contribute to retrieval quality, and
what (if anything) should change. Hypotheses were generated by a four-lens
research pass (chunking, model choice, field composition, evaluation design),
synthesized into a ranked experiment plan, then each experiment was run on a
purpose-built labeled corpus.

## TL;DR — ranked recommendations

> **Status:** recommendations #1 (metadata vector, `embeddings.metadata_vector`,
> default on) and #4 (runt-merge in `chunk_text`) are implemented.

| # | Change | Evidence | Cost |
|---|--------|----------|------|
| 1 | **Embed a metadata vector per memory** (`"{title}. {summary}. tags: …"` as an extra chunk row alongside the content chunks) — **implemented, default on** | MRR@10 0.75 → 0.89, P@1 0.67 → 0.85 on the default model; consistent on all 4 models tested (13 queries improved / 2 worsened). Tag-only queries go from MRR 0.15 to 0.78. | +1 vector per memory; a `reindex --embeddings-only`; no schema change (chunk table already holds N vectors, max-aggregation already handles it) |
| 2 | **Keep int8 MiniLM as the default model** — do not switch to BGE-small or nomic | BGE-small-q only matches MiniLM after adding its query prefix *and* the metadata vector (R@5 0.948 vs 0.920 — inside the noise floor) at 6.8× the latency (98.8 vs 14.6 ms/text). nomic-q int8 is far worse on every cell and 8.6× slower. | none (status quo) |
| 3 | **Keep 256-token chunks and max aggregation** | Budget sweep 64–256 is flat (±2 pts); mean aggregation *loses* up to 24 pts MRR on buried-fact queries; chunking beats no-chunking (truncation loses 5 pts R@5 overall, 14 pts on straddle-fact memories) | none (status quo) |
| 4 | **Merge trailing runt chunks** (<32 words into the previous chunk) — **implemented** | Small but clean: nDCG 0.780 → 0.817-run-1 / equal-or-better canonical, 5 improved / 0 worsened, and 7% *fewer* vectors | ~10 lines in `chunk_text` |
| 5 | **Token-aware chunk budget for code-dense text** (defect, mild today) | 25% of corpus chunks (50% of code-dense ones) exceed the 256-token model limit — up to 70 trailing tokens silently truncated. Recall impact currently mild (code-dense R@5 0.96) because queries rarely target only the truncated tail, but it's silent data loss that worsens with denser code. | tokenizer-based packing, or a smaller word-per-token factor when identifier density is high |
| 6 | Don't bother: query instruction prefixes for MiniLM, structured label prefixes (`type: … | scope: …`), sentence-aware or overlapping chunking, mean/top-2 aggregation | Each was tested; none produced gains above the noise floor, and several regress (details below) | — |

## Current implementation (what was measured against)

- Embedded text is exactly `format!("{summary} {content}")`
  (`src/retrieval/engine.rs::embed_memory_with`). **Title, tags, type, and
  logical scope never reach the embedding.**
- `chunk_text` (`crates/engram-models/src/embeddings/chunking.rs`) splits on
  whitespace into fixed blocks of `max_tokens * 3/4` words (192 words at the
  256-token default). No overlap, no sentence awareness.
- Each chunk is a row in the LanceDB chunk table; `vector_search` aggregates
  per-memory by **max** chunk score (`score = 1/(1+L2)`).
- Queries are embedded whole, with no instruction prefix (fastembed adds none).
- Default model: `AllMiniLML6V2Q` (int8, 384-dim, 256-token context).

## Method

**Corpus** (`examples/data/embed_eval.json`, frozen): 60 hand-authored
engineering memories across two fictional projects (Rust backend + TS web
app); 24 short / 19 medium / 17 long (350–500 words). Engineered strata:
planted facts at controlled positions vs the 192-word chunk boundary (4 start
/ 5 straddle / 8 end), 15 code-dense memories (env vars, paths, flags), 6
near-duplicate distractor pairs, 11 memories whose discriminating signal
exists **only** in title or tags. **Queries:** 48 with graded relevance
(2 = primary target, 1 = partial) across 7 archetypes: keyword, natural,
title_echo, tag_only, buried_fact, paraphrase, distractor_trap.

**Harness** (`examples/embed_matrix.rs`): mirrors store behavior (same
`chunk_text`, per-chunk vectors, max aggregation, whole-query embedding;
vectors L2-normalized so cosine ordering matches the store's `1/(1+L2)`).
16 document variants × {max, mean, top-2} aggregation × 4 models ×
{raw, prefixed} queries. Deterministic (sorted batch order). Semantic-only:
composite scoring (scope/recency/trust) and the reranker are *not* in the
loop, so effects here are embedding-level; composite scoring can only dilute,
not create, these deltas.

**Noise discipline** (from the eval-design lens): with n=48 queries, overall
deltas below ~5 pts R@5 / 0.05 MRR are treated as noise; per-archetype (n=6–12)
deltas need ~2 query flips to mean anything. Verdicts below follow that rule.

**Models** (all CPU, ONNX): `minilm-q` int8 384d (default), `minilm-fp32`,
`bge-small-en-v1.5-q` int8 384d, `nomic-embed-text-v1.5-q` int8 768d.

## Results by experiment

Overall numbers below are minilm-q, max aggregation, raw queries unless noted.

### E1 · Field composition — CONFIRMED, the dominant effect

| Variant | P@1 | R@5 | MRR@10 | nDCG@10 | win/loss |
|---|---|---|---|---|---|
| `base` (today) | 0.667 | 0.830 | 0.746 | 0.780 | — |
| `+title` | 0.729* | 0.899 | 0.818 | 0.837 | 9/5 |
| `+title+tags` (concat) | 0.771* | 0.906 | 0.873 | 0.880 | 11/3 |
| `labeled prefix` | 0.708* | 0.920 | 0.798 | 0.850 | 12/4 |
| **`fieldvec` (metadata as own vector)** | **0.854** | **0.920** | **0.894** | **0.904** | **13/2** |
| `ctx` (header on every chunk) | 0.812* | 0.868 | 0.835 | 0.851 | 12/4 |

*P@1 for intermediate variants from the canonical JSON; table rows share the run.*

Per-archetype MRR@10 shows *where* the gain lives, exactly as predicted:
tag_only 0.15 → 0.78, title_echo 0.64 → 1.00, while keyword/natural/buried-fact
stay flat (0.92 / 0.87→0.89 / 0.96→1.00) — i.e. **no dilution of content
matching**. The separate-vector form beats concatenation on precision
(P@1 +0.08 over `+title+tags`) because content chunks stay clean; the
structured label prefix (`type: … | scope: … | tags: …`) is strictly worse
than plain text tags (MRR 0.798 vs 0.873) — the negative control behaved as
predicted. This replicates across all four models (e.g. bge-small-q:
base MRR 0.776 → fieldvec 0.912; minilm-fp32: 0.772 → 0.933).

**Recommendation:** in `embed_memory_with`, emit one extra chunk text
`"{title}. {summary}. tags: {tags}"` ahead of the content chunks (skip when
title and tags are both empty — the summary is already in chunk 1). The chunk
table, max aggregation, freshness guard, and `has_embedding` maintenance all
work unchanged; existing stores pick it up via `engramdb reindex
--embeddings-only`. A config flag (`embeddings.metadata_vector = true`
default) keeps it reversible, and the provider cache key is unaffected.

### E2 · Model swap + instruction prefixes — REFUTED (keep MiniLM-q)

| Model | Variant | R@5 | MRR@10 | ms/text |
|---|---|---|---|---|
| minilm-q | base / fieldvec | 0.830 / 0.920 | 0.746 / 0.894 | 14.6 |
| minilm-fp32 | base / fieldvec | 0.819 / 0.931 | 0.772 / 0.933 | 26.0 |
| bge-small-q | base / fieldvec | 0.767 / 0.913 | 0.776 / 0.912 | 98.8 |
| bge-small-q | fieldvec + query prefix | **0.948** | 0.922 | 98.8 |
| nomic-q | best cell (fieldvec + prefixes) | 0.701 | 0.613 | 125.2 |

- The MTEB prior ("retrieval-tuned BGE-small should gain 5–10 pts over
  MiniLM at the same 384 dims") **did not transfer** to this workload: raw
  BGE-small is *worse* than MiniLM (R@5 0.767 vs 0.830), notably on tag-style
  keyword matching (tag_only R@5 0.08 vs 0.42). With its documented query
  prefix *and* the metadata vector it reaches R@5 0.948 vs MiniLM's 0.920 —
  a 2.8-pt edge, inside the noise floor, for 6.8× the latency.
- Prefix mechanics validated: the BGE prefix helps BGE (+0.03 MRR) but does
  nothing for MiniLM (0.746 → 0.737, the negative control), and nomic without
  its `search_document:`/`search_query:` prefixes collapses (MRR 0.363) —
  worth knowing if a nomic/Ollama backend is ever promoted: **prefixes are the
  caller's job** (fastembed adds none).
- nomic-q int8 underperformed everywhere (likely quantization damage on top of
  prefix sensitivity), and its 8192-token no-chunk mode was its *worst* cell
  (MRR 0.327) — long-context single-vector embedding is not a shortcut.
- int8 vs fp32 MiniLM: fp32 is slightly better on the composed variants
  (MRR 0.933 vs 0.894 on fieldvec) at 1.8× latency — consistent with the
  earlier Lever-B finding that int8 is the right default; revisit only if
  ranking precision ever becomes the bottleneck.

### E3 · Token-budget overflow — CONFIRMED as a defect, impact currently mild

True-token counts of `chunk_text` output (MiniLM tokenizer, truncation
disabled): **22/89 chunks (25%) exceed the 256-token model limit** — 14/28
(50%) of code-dense chunks, max 326 tokens (~70 tokens silently dropped by
fastembed). Prose: 8/61 (13%), mean 179.

Recall consequence today is small: code-dense R@5 is 0.96 at c256 (queries
usually match surviving text too), but full-text truncation (`base_trunc`)
drops code-dense R@5 to 0.82 — confirming that when enough text is cut,
recall follows. This is silent data loss that grows with identifier density;
fix cheaply by packing chunks against real token counts (the tokenizer is
already in-process) or dropping the `3/4` word factor to ~`0.6` for texts with
high identifier density.

### E4 · Chunk-budget sweep — REFUTED (256 is fine)

64→256-token budgets move overall metrics by ≤2 pts in both directions on
MiniLM (e.g. R@5 0.830 / 0.840 / 0.844 / 0.830 / 0.840 across 256→64); BGE
degrades at small budgets. No hump at 96–128 as hypothesized. Smaller budgets
also multiply vector count (251 vs 89 chunks at c64). Keep 256.

**Chunking vs no chunking:** truncated single-vector embedding loses 5 pts R@5
overall and 14 pts on straddle-fact memories — chunking is doing its job; the
question was never whether to chunk, only how.

### E5 · Boundary/overlap/runt fixes — mostly noise; runt-merge is a free win

Straddle-fact bucket (facts bisected by the 192-word boundary, n=5 queries):
base 0.79 R@5; 24-word overlap 0.86; 48-word overlap 0.79; sentence-aware
0.79; **contextual/fieldvec variants 0.93**. All single-query flips — below
the pre-registered per-stratum threshold, so: no overlap/sentence chunker is
justified by this data. Notably the metadata-vector change (E1) helps the
straddle bucket more than any chunker surgery.

Runt-merge (fold a trailing <32-word chunk into its predecessor): only
variant with a strictly-positive sign profile in run 1 (5 improved / 0
worsened, nDCG +0.012 canonical) while *reducing* index size 7%. Low-risk
`chunk_text` improvement.

### E6 · Aggregation — max validated

Mean aggregation loses 3.5 pts R@5 overall and 24 pts MRR on buried-fact
queries (0.96 → 0.72) — it dilutes exactly the multi-chunk memories chunking
exists to serve. Top-2-mean is a wash (+0.04 MRR on the baseline composition,
−0.01 on fieldvec). Keep max.

### E7 · Near-duplicate margin — INCONCLUSIVE (corpus too easy)

All MiniLM/BGE cells score P@1 1.00 on distractor-trap queries, so the twin
pairs didn't stress bi-encoder margins enough to test the reranker hypothesis.
(nomic-q at P@1 0.50 shows the failure mode exists.) Needs a harder twin set;
left open — the cross-encoder rerank path stays opt-in as today.

## Caveats

- Synthetic 60-memory corpus, n=48 queries: effects below ~5 pts are noise;
  the headline E1 effect (+15 pts MRR, 13/2 sign profile, replicated on 4
  models) is far above it, but treat small deltas as directional only.
- Semantic-only harness: composite scoring (scope, recency, trust, lexical
  keyword boost) will shrink observed end-to-end deltas; it cannot create a
  blind spot where the vector never contained the signal — which is exactly
  what the metadata vector fixes.
- int8 dynamic quantization is batch-composition-sensitive (~±0.02 P@1 before
  the sorted-batch fix); rankings, not absolute scores, are the stable output.
- `title` is T5/keyword-generated at create time *before* embedding
  (`ops/create.rs` resolves title, then embeds), so the metadata vector can
  use it with no ordering change; it degrades gracefully when
  `title.strategy = "none"` (falls back to summary+tags).

## Reproducing

```bash
# stage models per docs (web sandbox: see CLAUDE.md workarounds), then:
cargo run --release --example embed_matrix
EMBED_EVAL_MODELS=minilm-q cargo run --release --example embed_matrix   # single model
EMBED_EVAL_OUT=/tmp/results.json cargo run --release --example embed_matrix
```

The committed `examples/data/embed_matrix_results.json` is the canonical
snapshot this report cites (deterministic modulo ONNX thread scheduling).
